# Этап 2 — Доменная модель GTD (совместимый переход)

## Цель
Постепенно перейти от текущих `Task`/`Column` к GTD-модели без «big bang» рефакторинга и без поломки уже работающего UI.

## Зависимости
- Требует завершения `tasks/1.md`.

## Checklist

### data
- [ ] Создать `src/types/gtd.ts` с типом `Item` (raw capture): `id`, `title`, `notes`, `createdAt`, `source`, `clarified`.
- [ ] Создать в `src/types/gtd.ts` enum/union для `ClarifyOutcome`: `next_action | project | someday | trash`.
- [ ] Создать в `src/types/gtd.ts` тип `NextAction` с полями MVP-ready (`id`, `title`, `contextId`, `status`, `projectId?`).
- [ ] Создать в `src/types/gtd.ts` тип `Project` с полями MVP-ready (`id`, `title`, `status`, `createdAt`, `reviewAt`).
- [ ] Создать тип `Context` в `src/types/gtd.ts`.
- [ ] Добавить список дефолтных контекстов в `src/constants/contexts.ts` (`@computer`, `@phone`, `@home`, `@deep-work`, `@5min`).
- [ ] Добавить в `src/types/storage.ts` следующую версию snapshot для GTD-сущностей.

### state
- [ ] Добавить в `src/hooks/useBoardState.ts` отдельные состояния для `items`, `nextActions`, `projects`, `contexts`.
- [ ] Добавить функцию маппинга legacy `tasks` -> `items` для мягкой миграции.
- [ ] Добавить флаг миграции `isMigratedFromLegacy` в snapshot (если нужен один раз).
- [ ] Добавить безопасный одноразовый запуск миграции при загрузке.
- [ ] Добавить селектор неуточненных элементов (`inboxItems` = `clarified === false`).
- [ ] Добавить селектор активных next actions (`status === active`).
- [ ] Добавить селектор активных проектов.

### ui
- [ ] Добавить в `src/components/BoardHeader.tsx` визуальный счетчик неуточненных Inbox items.
- [ ] Обновить текстовые label в `src/components/BoardColumn.tsx`: «задача» -> «item», где это не ломает UX.
- [ ] Добавить временный бейдж `Legacy` для элементов, пришедших из старых `tasks` (до полного перехода).

### validation
- [ ] Проверить миграцию на пустом state (новый пользователь).
- [ ] Проверить миграцию на непустом state (есть старые колонки и задачи).
- [ ] Проверить, что после миграции не пропадают id/createdAt.
- [ ] Проверить, что старый workflow (добавить/перетащить/удалить) не ломается до финального отключения legacy.

### manual-test
- [ ] Подготовить state в старом формате, перезапустить приложение, убедиться в успешной миграции.
- [ ] Открыть приложение дважды подряд (два reload), убедиться что повторная миграция не искажает данные.
- [ ] Проверить, что Inbox count отображает количество `clarified === false`.

## Definition of Done
- Добавлены базовые GTD-типы и дефолтные контексты.
- В состоянии доступны GTD-сущности и селекторы.
- Legacy-данные мягко мигрируются и не теряются.
- Приложение остается работоспособным на переходном этапе.
